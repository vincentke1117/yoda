# Yoda · 致每一个被 Agent 折磨过的开发者（宣言版）

我知道，你们很累。又一个 coding agent 发布了，又得重配一遍环境、重学一套命令。我累的，不是这个。AI 三年来，模型一代比一代聪明，可干起活来，我们还是回到"开十个终端、切八个窗口、记不清哪个会话在跑什么"的老样子。这问题不解决，agent 再聪明，也只是又一个让你疲于奔命的黑盒。

那么，我们真正想要的工作区，究竟该是什么样子？

**它应该是统一的。** 可这些年我们看到的是什么？Claude Code 开一个终端、Codex 开一个终端、Gemini 又开一个终端，同一件事在七八个窗口里各说各话。Yoda 把 27 种 Runtime 收进同一个工作区——按任务挑选、随时恢复会话、跑对比、互相 review；Linear、Jira、GitHub 的工单直接落进会话，CI/CD 状态就摆在 diff 旁边。一个工作区，而不是一地鸡毛。

**它应该是自由的。** 可这些年我们看到的是什么？工具替你定死了一套范式：要么逼你处处建分支、开 worktree，要么只给一条直线，不容并行、不容对比。Yoda 把"用哪种范式"还给你：想快，主分支直跑，开任务、让 AI 干活、归档收尾，归档前还能自动跑你预设的 Skill；想稳，每个任务一个独立 worktree，并行互不干扰，审完 diff 再合并；想深，还有 brainstorm、compare、review、team 随你切换。是你挑工作流，不是工作流挑你——换 CLI，也不必重写你的 Agent 资产。

**它应该是透明的。** 人人都说"提示词为王"，可提示词只是水面上的一层；agent 真正的行为，是它加载的 Skills、Hooks、Memory 和上下文决定的。可这些年我们看到的是什么？出了问题，你只能对着黑盒猜：是模型变笨了，还是我哪里没写对？Yoda 把这条链路整个摊开——逐项查看、按需覆写、持久化。多数"agent 怎么突然不行了"，根子不在模型，就在这层没人给你看的 harness 里。

这就是说——没有"把 harness 当作产品"的观念深入人心，就不可能有真正好用的 agent 工作流。它不该再是一个被忽略的术语、一句"模型决定一切"的空话，而该成为你每天实实在在的工作方式，成为你和你那队 agent 之间牢不可破的默契。

这，就是 Yoda：**Your Orchestra of Delegated Agents**——你那支被妥善指挥的 agent 乐队。我深信，总有一天，"指挥一队 agent 干活"会像今天打开编辑器一样天经地义。而这一天，不必再等。

不绑定模型，不绑定厂商，本地优先。今天就把它请回你的工作台：
