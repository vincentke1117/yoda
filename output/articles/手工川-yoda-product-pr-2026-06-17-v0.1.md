# Yoda · 致每一个被 Agent 折磨过的开发者（宣言版）

我知道，你们很累。又一个新的 coding agent 发布了，又得重新配一遍环境，又得重新学一套命令，我知道。我呀，我累的不是这个。这些日子我想了很多——我们手里明明已经有了最聪明的模型，可为什么干起活来，还是一次又一次地回到"开十个终端、切八个窗口、记不清哪个会话在跑什么"的老样子呢？这个问题不解决，agent 再聪明，也只是又一个让你疲于奔命的黑盒；所谓"AI 提效"，就永远是一个泡影。

那么，我们真正想要的那个工作区，应该是个什么样子呢？

**它应该是统一的。** 编排的观念，本就是"工具为人所用"，而不是"人被工具撕成碎片"。可这些年我们看到的是什么？是 Claude Code 开一个终端、Codex 开一个终端、Gemini 又开一个终端；是同一件事，在七八个窗口里各说各话。Yoda 把 27 种 Runtime 收进同一个工作区——按任务挑选、会话随时恢复、跑对比、互相 review；Linear、Jira、GitHub 的工单直接落进会话，CI/CD 的状态就摆在 diff 旁边。一个工作区，而不是一地鸡毛。

**它应该是自由的。** 自由，是开发者天赋的权利——而开发者最该有的，是选择"怎么干"的自由。可这些年我们看到的是什么？是工具替你定死了一套范式：要么逼你处处建分支、开 worktree，要么干脆只给你一条直线，不容你并行、不容你对比。Yoda 把"用哪种范式"这件事还给你自己：想快，就主分支直跑，不建分支、不开 worktree，开任务、让 AI 干活、归档收尾，归档前还能自动跑你预设的 Skill 替你把活收干净；想稳，就每个任务一个独立 git worktree，并行互不干扰，审完 diff 再合并；想深一点，还有 brainstorm、compare、review、team 几种模式随你切换。是你来挑工作流，不是工作流挑你；换 CLI，也不用重写你的 Agent 资产。

**它应该是透明的。** 人人都说"提示词为王"，可提示词只是浮在水面上的那一层。agent 真正的行为，是被它加载的 Skills、Hooks、Memory 和上下文决定的。可这些年我们看到的是什么？是只有口头上的"可解释"——出了问题，你只能对着一个黑盒猜：是模型变笨了，还是我哪里没写对？Yoda 把这条链路整个摊开在你面前：逐项查看、按需覆写、持久化。多数"agent 怎么突然不行了"的毛病，根子不在模型，就在这层没人给你看的 harness 里。

这就是说——没有"把 harness 当作产品"的观念深入人心，就不可能有真正好用的 agent 工作流。

所以我们要让 harness——不再仅仅是一个被忽略的术语、一句"模型决定一切"的空话、一个开十个终端的将就形式，而要让它成为你每天实实在在的工作方式，成为你和你那一队 agent 之间，牢不可破的默契。

这，就是 Yoda：**Your Orchestra of Delegated Agents**——你那支被妥善指挥的 agent 乐队。

不绑定模型，不绑定厂商，本地优先。今天就把它请回你的工作台：
