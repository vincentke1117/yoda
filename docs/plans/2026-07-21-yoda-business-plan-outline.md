# Yoda 商业计划书（BP）大纲

> 版本：v0.2（2026-07-21）
> 阅读对象：投资人，不预设读者具备 AI、编程或开发者工具背景。
> 写作原则：已验证事实与商业假设分开；所有市场、增长、收入数据均注明口径和日期；执行摘要最后写。

## 一句话定位

**让一个开发者指挥一支 AI 开发团队。**

下一层解释：Yoda 是超级开发者的 AI 研发指挥台。它通过 Harness 管理 Agent 的会话、Skills、Hooks、Memory、Diff Review 与 CI/CD，让开发者更顺畅地驾驭 AI，自由创作和交付想做的 App。

## 0. 使用方式与建议篇幅

- 正式商业计划书建议控制在 15–20 页，附录单列产品截图、数据口径和财务模型。

- 本大纲中的“已验证”仅表示可由当前仓库材料支持，不代表市场或经营数据已经审计。
- 标记为“待验证”的内容在进入正式 BP 前必须补充来源、计算过程或负责人确认。

## 1. 执行摘要（最后撰写，正式稿 1 页）

严格压缩为五个要素：

1. **Yoda 做什么**：Yoda 是超级开发者的 AI 研发指挥台。它以 Harness 为核心，把主流 Agent Client、会话、Skills、Hooks、Memory、代码审查和发布流程放进同一个工作空间，让开发者可以用一个或多个 Agent 创作并交付 App。
2. **为谁解决什么问题**：服务独立开发者、技术创始人和小型 AI 原生团队。用户即使只使用一个 Agent，也会遇到工具和状态碎片化、Harness 资产分散、恢复上下文麻烦，以及需要在多个窗口持续盯住开发过程的问题。
3. **解决方案与差异**：Yoda 不绑定某个模型或厂商，支持单 Agent、Vibe Coding、多智能体协作和 Build 等开发范式；它不试图消灭人的判断和盯守，而是让人更容易看清状态、干预过程、审查结果并持续迭代。
4. **市场机会**：全球约 4,840 万开发者，84% 已使用或计划使用 AI 开发工具。按个人基础套餐 120 美元/年测算，TAM 约 48.8 亿美元，当前可服务市场约 7.32 亿美元，三年可获取市场目标约 240 万美元 ARR；计算口径见市场分析。
5. **当前诉求**：融资 200 万元人民币或 30 万美元，出让 10% 股权；资金主要用于技术研发和市场推广，支撑未来 18–24 个月产品完善、增长验证与商业化。

- 使命：让每个有创造力的开发者，都能顺畅驾驭 AI 完成真实产品。
- 愿景：让一个开发者指挥一支 AI 开发团队。
- 组织策略：以一人公司/小团队为起点，通过 Agent、自动化、开源社区和生态伙伴放大交付能力。

## 3. 问题、解决方案与 Why Now

### 3.1 核心问题

围绕一个主问题展开，而不是罗列功能：AI 已经会写越来越多代码，但开发者仍缺少一个友好的指挥台去持续组织、观察、干预和验收 AI 的工作。

建议用三组可量化痛点支撑：

1. **工具与状态碎片化**：使用一个 Agent 时，开发者已经需要在终端、IDE、网页、代码仓库和发布工具之间切换；Agent 增多后，状态更加分散。
2. **Harness 资产难管理**：Prompt、Skills、Hooks、Memory、模型配置和权限并非不可见，而是散落在不同客户端与文件中，查看、迁移、复用和调试都很麻烦。
3. **人工盯守体验不友好**：工程交付在可预见的未来仍需要人做判断、干预和验收。问题不是“还要人盯”，而是人必须在多个窗口低效轮询，难以快速理解进度和风险。

正式稿需补：

- 10–20 位目标用户访谈中的原话、频率和角色。
- 从任务启动到合并的时间分布，以及使用 Yoda 前后的对照数据。
- 使用单 Agent 与多 Agent 时发生上下文重述、会话丢失、反复切换窗口和人工等待的频率。

### 3.2 Yoda 的解决方案

先展示多种开发范式，再描述 Yoda 如何用统一 Harness 承接它们。分支或 Worktree 只是可选的隔离开关，不作为核心卖点：

```text
一个想法 / 一项任务
  → 选择单 Agent、Vibe Coding、多智能体协作或 Build 模式
  → 按需要选择模型、MaaS 平台与隔离方式
  → Yoda 统一管理 Session、Skills、Hooks、Memory 与执行状态
  → 开发者随时观察、干预、比较和 Review
  → Diff、测试与 CI/CD 验证
  → 继续迭代、发布或沉淀为可复用 App
```

已可由仓库材料支持的能力：

- 统一接入 31 种 Agent Client；既可只使用一个 Agent，也可按任务组合多个 Agent。
- 覆盖普通开发、Vibe Coding、brainstorm、compare、review、team 和 Build 等多种开发范式。
- 自定义 Agent、Skills、Hooks、Memory 与会话上下文观测。
- Git Worktree 隔离、Diff/Review、工单接入和 CI/CD 状态联动；是否启用隔离由用户按任务决定。
- 本地与云端混合架构：项目与代码默认留在用户设备，模型、MaaS、账号、云同步和 Relay 按需使用云服务。这里的“本地”指数据与执行边界，不代表大模型在本机运行。
- Mobile/Gateway、Relay、SSH 远程项目与跨平台安装包，支持跨设备继续观察和干预 Agent 工作。
- Yoda Build：从自然语言启动构建任务，并把生成的子 App 保存到 Library / Apps 继续迭代。

### 3.3 Before / After

| 环节 | 使用前 | 使用 Yoda 后 | 正式稿指标 |
| --- | --- | --- | --- |
| 开始开发 | 在不同工具里分别启动和配置 Agent | 从统一入口选择适合的开发范式 | 首次价值时间、任务启动时间 |
| 过程盯守 | 在多个窗口反复轮询、难以判断下一步 | 在一个指挥台观察状态并随时干预 | 切换次数、人工等待时间 |
| 上下文恢复 | 跨工具重新解释背景 | 会话、Skills、Memory 与交付摘要关联 | 恢复时间、重述次数 |
| 质量验证 | 人工轮询输出与测试 | Diff、Review、测试和 CI 状态进入同一闭环 | 缺陷率、审查时间 |
| 创作与迭代 | 代码生成后仍需手工拼接工作流 | 从想法、App 构建、审查到继续迭代保持连续 | 交付周期、二次迭代率 |

### 3.4 Why Now

- 编码 Agent 从代码补全演进为可承担完整任务的执行者，开发者的角色正从“亲自写每行代码”转向“提出目标、提供上下文、判断和验收”。
- Claude Code、Codex、Gemini 等客户端快速增加，开发者需要供应商中立的控制层，而不是再次绑定单一模型。
- Agent 的差异越来越取决于 Harness：上下文、工具、Skills、Hooks、Memory、权限和交付流程。
- [2025 Stack Overflow 开发者调查](https://survey.stackoverflow.co/2025/ai)显示，84% 的受访者已使用或计划使用 AI 开发工具，但 87% 仍担心准确性、81% 担心安全与隐私，说明人的观察、干预和验证不会消失。

以上趋势必须在正式稿中补充近 12 个月的一手或权威来源。

## 4. 市场分析

### 4.1 目标客户

优先聚焦两个细分群体：

1. **独立开发者与技术创始人**：使用一个或多个编码 Agent，希望用更少人力完成从想法到发布的完整交付。
2. **2–20 人 AI 原生研发团队**：需要统一 Agent 工作流、权限、审查、远程接入与可复用组织资产。

次级客户可包括 Agent 咨询/交付团队、开发者工具团队和对本地部署敏感的专业组织；正式 BP 不应同时把所有开发者都列为首要客户。

### 4.2 TAM / SAM / SOM 基准测算

| 市场 | 基准数字 | 计算方式 | 口径说明 |
| --- | ---: | --- | --- |
| TAM | 48.8 亿美元/年 | 4,840 万全球开发者 × 84% AI 工具使用/计划使用率 × 120 美元年费 | 开发者人数来自 SlashData Q3 2025；AI 采用率来自 Stack Overflow 2025 |
| SAM | 7.32 亿美元/年 | 4,840 万 × 84% × 15% 目标用户适配率 × 120 美元年费 | 15% 是对独立开发者、技术创始人和 AI 原生小团队的待验证假设 |
| SOM（3 年） | 240 万美元 ARR | 2 万个付费账户 × 120 美元年费 | 是经营目标，不是当前收入或市场份额事实 |

- [SlashData](https://www.slashdata.co/research/developer-population) 的最新公开估计为全球 4,840 万活跃开发者；[GitHub Octoverse 2025](https://github.blog/news-insights/octoverse/octoverse-a-new-developer-joins-github-every-second-as-ai-leads-typescript-to-1/) 的 1.8 亿账户规模只用于说明生态触达面，不与活跃开发者口径混算。
- 120 美元年费与当前 Relay 990 credits/30 天（约 9.9 美元/月）的价格锚点接近，正式模型仍需区分免费版、Pro、Relay 和 Team。
- SAM 中 15% 的目标适配率必须通过用户研究、定价测试和平台数据继续缩小误差；不能把 TAM 直接当收入机会。

### 4.3 市场趋势

- Agentic coding 从代码补全转向任务级、仓库级和团队级执行。
- 开发者开始组合多个模型与客户端，单一厂商锁定与资产迁移成为痛点。
- 本地数据边界与云端能力并存；代码隐私、过程可审计和跨设备协作共同成为购买条件。
- Skills、Hooks、Memory 和 Agent 工作流开始形成可复用、可分发的生态资产。

## 5. 竞争格局与竞争楔子

### 5.1 建议比较的四类方案

- 单一厂商 Agent IDE / CLI。
- 云端自主编码 Agent。
- 终端复用、脚本和手工 Worktree 流程。
- 多 Agent 编排框架或工作台。

正式稿选取 3–5 个真实竞品，按下列维度建立矩阵：供应商中立、代码数据边界、单/多 Agent、开发范式、会话恢复、Harness 可观测、Review/CI 闭环、移动/远程、团队治理、价格。

### 5.2 Yoda 的竞争楔子

- **供应商中立**：统一承载不同 Agent CLI，用户可按任务选择工具并保留资产。
- **Harness 可观测**：不只管理聊天窗口，而是暴露 Skills、Hooks、Memory 和上下文链路。
- **真实工程闭环**：把 Issue、会话、Worktree、Diff、Review、CI、合并和归档连接起来。
- **本地与云端混合架构**：项目和代码默认留在用户设备，模型、MaaS、同步与 Relay 等云能力按需接入，边界清楚且不误导为“本地大模型”。
- **兼容个人效率与团队治理**：先服务一人公司的高频工作流，再向小团队权限与协作扩展。

### 5.3 诚实披露的竞品优势

- 单一厂商产品可能拥有更深的模型集成、默认体验和资金优势。
- 云端 Agent 可能拥有更强的托管算力和无人值守体验。
- IDE 类产品拥有成熟编辑器生态和庞大分发渠道。
- Yoda 需要证明跨客户端编排带来的价值足以抵消配置和学习成本。

## 6. 商业模式与收入模型

### 6.1 建议商业路径

```text
开源 Yoda Desktop 建立信任与获客
  → Relay / 云同步等个人增值能力
  → 团队席位、权限、审计与共享 Harness 资产
  → Skills / 工作流 Marketplace 与托管执行能力
```

### 6.2 收入流（按验证顺序）

1. **Relay Pass / 个人订阅**：跨网络安全连接桌面 Agent，验证个人用户持续付费意愿。
2. **Pro 云能力**：云同步、跨设备体验、增强自动化和历史保留；范围与价格待验证。
3. **Team 席位**：团队权限、共享 Agent/Skills/Memory、审计、策略与协作工作流。
4. **Marketplace 抽成**：第三方 Skills、模板和工作流交易；必须在供需两侧形成规模后启动。
5. **企业支持或私有化服务**：作为后期收入补充，避免早期被重交付拖入项目制。

### 6.3 底层收入公式

- 月新增付费用户 = 有效触达 × 安装率 × 激活率 × 试用转付费率。
- 月经常性收入 = 各套餐活跃付费用户 × 对应月均客单价。
- LTV = 月均毛利 × 平均付费生命周期。
- CAC = 获客相关现金支出 ÷ 同期新增付费用户。
- 回本周期 = CAC ÷ 单用户月均毛利。

正式稿需明确：免费与付费边界、套餐价格、退款/续费规则、支付成本、Relay 基础设施成本和开源核心不商业化的边界。

## 7. 运营计划

### 7.1 产品交付

- Desktop 通过 GitHub Releases 和官网安装包分发。
- 版本更新、签名、公证、跨平台构建和镜像分发由 CI/CD 承担。
- 用户代码与本地状态默认留在本机；Relay、账号和云同步明确区分数据边界。
- 文档、Issue、更新日志和社区内容共同承担上手、支持与反馈收集。

### 7.2 一人公司关键流程

1. 用户反馈与问题归类。
2. Feature → Task → Agent 执行 → Review → Verification → Release。
3. 跨平台打包、更新源与公开资产验证。
4. Relay/云服务可用性、成本与安全监控。
5. 内容发布、案例沉淀和用户访谈。

### 7.3 自动化与外包边界

- 优先自动化测试、构建、发布、更新源验证、支持分类和经营数据看板。
- 当支持工时、平台兼容性或内容生产超过预设阈值时，引入兼职/外包伙伴。
- 产品方向、架构、安全边界和关键用户访谈继续由创始人直接负责。

## 8. 营销与销售计划

### 8.1 定位语

**让一个开发者指挥一支 AI 开发团队。** Yoda 通过统一 Harness，把分散的 Agent、上下文资产和开发流程组织成一个可以持续观察、干预和验收的 AI 研发指挥台。

### 8.2 优先渠道

1. **开源与开发者社区**：GitHub、Release、Issue、真实用户案例和可复现工作流。
2. **创始人内容**：围绕 Harness Engineering、一人公司、多 Agent 交付和 Build in Public 持续输出。
3. **生态合作**：Agent CLI、MaaS、模型平台、Skill 作者和开发者社区的联合内容与集成。

### 8.3 销售漏斗

```text
内容/社区触达
  → 官网或 GitHub
  → 下载与首次启动
  → 完成首个真实任务
  → 7 日内完成第二个任务或启用多 Agent/远程能力
  → Relay/Pro 试用
  → 付费与续费
```

正式稿为每一层补充当前基线、12 个月目标、责任渠道与实验计划。北极星指标建议从“每周完成并验证的 Agent 交付任务数”中验证，而不是只看下载或 Stars。

## 9. 财务预测

### 9.1 模型结构

- 第 1 年按月，第 2–3 年按季度。
- 至少包含保守和基准情景；乐观情景单列，不混入经营承诺。
- 收入：Relay/Pro、Team、Marketplace、企业支持分别建模。
- 成本：支付手续费、Relay/云资源、监控、代码签名/发布、SaaS 工具、市场投放、外包和人员。
- 输出：收入、直接成本、毛利、运营费用、净利润/亏损和期末现金。

### 9.2 必算阈值

- 月度盈亏平衡所需付费用户数。
- Relay/云能力的单用户边际成本与毛利率。
- 创始人投入和现金储备对应的可持续月数。
- 在付费转化、续费率或基础设施成本恶化时的现金流敏感性。

## 10. 风险评估

正式稿按“概率 × 影响 × 缓解措施 × 预警指标”维护：

| 风险 | 概率 | 影响 | 缓解措施 | 预警指标 |
| --- | --- | --- | --- | --- |
| 单一创始人依赖与倦怠 | 中高 | 高 | 自动化发布与支持；关键流程文档化；建立兼职伙伴池 | 交付延期、支持积压、连续高工时 |
| Agent CLI/API 高频变化 | 高 | 高 | 可扩展 registry；兼容性测试；按使用量划分维护优先级 | 集成故障率、修复时长 |
| 大厂把编排能力内置 | 高 | 高 | 坚持供应商中立、清晰数据边界和跨客户端 Harness 资产层 | 跨客户端用户占比下降 |
| 用户认可产品但不付费 | 中高 | 高 | 先验证 Relay/Pro 的刚需与续费，再扩展商业范围 | 试用转付费、续费率低 |
| 开源与付费边界失衡 | 中 | 高 | 明确核心开源、云连接/团队治理付费的原则 | 社区反弹、付费功能使用低 |
| 安全、隐私或供应链事件 | 中 | 高 | 最小权限、本地默认、签名公证、依赖审计、事件响应流程 | 异常访问、签名/更新失败 |
| 多平台与 31 客户端维护成本过高 | 高 | 中高 | 按活跃使用分级支持；自动化回归；社区共建 | 平台缺陷积压、兼容性 SLA 下降 |
| Marketplace 冷启动 | 中高 | 中 | 在交易前先做可分享目录、官方模板和作者激励 | 有效供给、安装与复用率低 |

## 11. 18–24 个月季度里程碑

以下是投资人版本的建议目标，用户数和收入目标需与完整财务模型再次校准：

| 时间 | 核心目标 | 关键交付与验证 |
| --- | --- | --- |
| 0–3 个月 | 产品功能完善、体验顺滑 | 打磨单 Agent、Vibe Coding、多智能体与 Build 主路径；建立崩溃、激活、任务完成、DAU、D7/D30 留存埋点；完成首轮 20 位目标用户访谈 |
| 4–6 个月 | 建立可重复的自然增长 | 完善官网、文档、上手模板和案例；开始持续内容推广与社区合作；验证“首次完成真实任务”激活漏斗 |
| 7–9 个月 | 开始小规模投放与付费验证 | 基于前两季度数据逐步投放；迭代 Relay/Pro 定价；验证试用转付费、续费和单用户云成本 |
| 10–12 个月 | 找到 2–3 个高留存场景 | 聚焦可传播的核心场景；建立 cohort 看板和自动化运营；让增长、激活与付费形成可复现漏斗 |
| 13–15 个月 | 验证小团队需求 | 与设计伙伴测试共享 Agent、Skills、Memory、权限和审计；形成 Team 版本最小闭环 |
| 16–18 个月 | 扩大市场推广 | 加大有效渠道投放和生态合作；扩展英文市场；验证个人订阅与 Team 席位的组合收入 |
| 19–21 个月 | 建立 Harness 生态 | 完善 Skill/工作流的发布、信任、安全扫描和发现；先验证供给、安装与复用，再决定交易抽成 |
| 22–24 个月 | 证明规模化能力 | 形成能够冲击 2 万付费账户、240 万美元 ARR 三年 SOM 的增长模型；验证增长效率、毛利和组织可持续性是否支持下一轮融资 |

每季度设置继续、调整或停止的量化门槛。前三个月不急于大规模投放，先把产品体验和数据基础做扎实；之后再根据真实留存逐步增加推广预算。
