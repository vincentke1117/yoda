# Yoda 商业计划书（BP）大纲

> 版本：v0.8（2026-07-22）
> 阅读对象：投资人，不预设读者具备 AI、编程或开发者工具背景。
> 写作原则：已验证事实与商业假设分开；所有市场、增长、收入数据均注明口径和日期；执行摘要最后写。

## 一句话定位

**Yoda：超级开发者的 AI Harness 工作台。**

**统一 Agent、上下文与交付，让 AI 真正进入持续创作。**

下一层解释：Harness 是一套帮助人驾驭和管理 AI 的工作系统。Yoda 把单个 Agent、多个 Agent、Skills、Hooks、Memory、Review 和发布流程组织成可观察、可控制、可复用的创作环境，并从超级开发者向内容创作者、广义 Coding 用户及其他 AI 原生个人与团队辐射。

## 0. 使用方式与建议篇幅

- 正式商业计划书建议控制在 15–20 页，附录单列产品截图、数据口径和财务模型。

- 本大纲中的“已验证”仅表示可由当前仓库材料支持，不代表市场或经营数据已经审计。
- 标记为“待验证”的内容在进入正式 BP 前必须补充来源、计算过程或负责人确认。

## 1. 执行摘要（最后撰写，正式稿 1 页）

严格压缩为五个要素：

1. **Yoda 做什么**：Yoda 是超级开发者的 AI Harness 工作台。它统一 Agent、上下文与交付，让 AI 不只生成一次结果，而是持续参与从想法、执行、审查到交付的完整创造过程。
2. **为谁解决什么问题**：核心用户是能够借助 AI 完成过去需要多人协作成果的“超级开发者”，并向内容创作者、广义 Coding 用户以及其他 AI 原生个人与团队辐射；同时，高校、培训机构和企业研发/学习发展部门也需要一套统一、可控、可教学的 AI 工作环境。他们共同需要的不是更多 AI 对话框，而是一套更好地组织、控制和复用 AI 能力的工作系统。
3. **解决方案与差异**：Yoda 不限定用户必须使用几个 Agent。单 Agent、多 Agent 和 Agent 团队都是可按任务选择的工作方式；真正的产品核心是 Harness——让人持续掌握目标、上下文、过程、质量与最终决定，并把每次创作沉淀为个人或组织可复用的资产。开源、可扩展的底座使机构能够在 Yoda 上配置自己的模型、课程、工作流和治理边界。
4. **市场机会**：Visa 2025 报告估计全球约有 2.07 亿创作者；按每年 120 美元的基础工具支出测算，理论 TAM 为 248 亿美元。Yoda 先从超级开发者和 AI 原生创作者切入，再向更广泛的创作者与团队扩展；当前 SAM 与三年 SOM 的计算口径见市场分析。
5. **当前诉求**：融资 200 万元人民币或 30 万美元，出让 10% 股权；资金主要用于技术研发和市场推广，支撑未来 18–24 个月产品完善、增长验证与商业化。

## 3. 问题、解决方案与 Why Now

### 3.1 核心问题

围绕一个主问题展开，而不是罗列功能：大模型已经显著提高了生成代码和内容的能力，但“能生成”不等于“能创造”。要把想法持续变成可用、可迭代、可发布的作品，人仍需要组织上下文、选择工具、观察过程、判断质量并积累方法。

建议用三组可量化痛点支撑：

1. **AI 能力强，但难以驾驭**：模型、Agent 和客户端不断增加，创作者需要反复选择、配置和切换工具，却缺少统一的控制与反馈界面。
2. **创作上下文与方法难沉淀**：Prompt、Skills、Hooks、Memory、模型配置和权限散落在不同客户端与文件中，成功经验难以观察、组合、复用和迁移。
3. **人工参与方式不友好**：创造过程在可预见的未来仍需要人提供品味、判断、干预和验收。问题不是“还要人参与”，而是人必须在多个窗口低效轮询，难以快速理解进度和风险。

正式稿需补：

- 10–20 位不同类型用户访谈中的原话、频率和角色，至少覆盖超级开发者、内容创作者、广义 Coding 用户和非 Coding 的 AI 原生用户。
- 从任务启动到合并的时间分布，以及使用 Yoda 前后的对照数据。
- 使用单 Agent 与多 Agent 时发生上下文重述、会话丢失、反复切换窗口和人工等待的频率。

### 3.2 Yoda 的解决方案

先展示用户如何完成创作，再描述 Yoda 如何用统一 Harness 承接不同工作方式。单 Agent、多 Agent、Agent 团队、分支或 Worktree 都是按任务选择的方式或开关，不作为产品定位：

```text
一个想法 / 一个创作目标
  → 选择单 Agent、多 Agent、Agent 团队、Vibe Coding 或 Build 模式
  → 按需要选择模型、MaaS 平台与隔离方式
  → Yoda 统一管理 Session、Skills、Hooks、Memory 与执行状态
  → 创作者随时观察、干预、比较和 Review
  → Diff、测试与 CI/CD 验证
  → 继续迭代、发布或沉淀为可复用的 App、产品与工作流
```

已可由仓库材料支持的能力：

- 统一接入 31 种 Agent Client；既可只使用一个 Agent，也可按任务组合多个 Agent。
- 覆盖单 Agent、Vibe Coding、brainstorm、compare、review、team 和 Build 等多种创作与开发范式。
- 自定义 Agent、Skills、Hooks、Memory 与会话上下文观测。
- Git Worktree 隔离、Diff/Review、工单接入和 CI/CD 状态联动；是否启用隔离由用户按任务决定。
- 本地与云端混合架构：项目与代码默认留在用户设备，模型、MaaS、账号、云同步和 Relay 按需使用云服务。这里的“本地”指数据与执行边界，不代表大模型在本机运行。
- Mobile/Gateway、Relay、SSH 远程项目与跨平台安装包，支持跨设备继续观察和干预 Agent 工作。
- Yoda Build：从自然语言启动构建任务，并把生成的子 App 保存到 Library / Apps 继续迭代。

### 3.3 Before / After

| 环节 | 使用前 | 使用 Yoda 后 | 正式稿指标 |
| --- | --- | --- | --- |
| 开始创造 | 先学习和拼接不同技术工具 | 从一个想法直接选择适合的 Agent 创作方式 | 首次价值时间、创作启动时间 |
| 过程盯守 | 在多个窗口反复轮询、难以判断下一步 | 在一个指挥台观察状态并随时干预 | 切换次数、人工等待时间 |
| 上下文恢复 | 跨工具重新解释背景 | 会话、Skills、Memory 与交付摘要关联 | 恢复时间、重述次数 |
| 质量验证 | 人工轮询输出与测试 | Diff、Review、测试和 CI 状态进入同一闭环 | 缺陷率、审查时间 |
| 创作与迭代 | AI 生成一次结果后，仍需手工拼接后续流程 | 从想法、生成、审查到发布和继续迭代保持连续 | 交付周期、二次迭代率 |

正式 BP 不只放表格，应增加一个 30–60 秒的典型场景：同一个用户、同一项任务，使用前需要在终端、IDE、Agent 客户端、Git 和 CI 页面之间来回切换；使用后从 Yoda 进入任务，在一个工作区完成启动、观察、干预、Review 和验证。是否能压缩到“一次点击”必须以真实操作录像和耗时数据为准，不能先写宣传数字。

### 3.4 Why Now

- Codex 等 Agent 从代码补全演进为可承担完整任务的执行者，人的角色正从“亲自完成每一步”转向“提出目标、提供上下文、驾驭过程、判断和验收”。
- GPT、Claude、Codex 等模型和 Agent 同时降低了代码、产品与内容创作门槛，创作能力开始从专业团队扩散到每一个有想法的人。
- Agent 的差异越来越取决于 Harness：上下文、工具、Skills、Hooks、Memory、权限和交付流程。
- 模型能力正在逐步商品化，不同模型和客户端的可替换性提高；当推理成本下降、长任务和并行执行变得可承受时，价值会进一步从模型本身转移到“如何组织、控制和验证 AI 工作”。正式 BP 需用模型厂商的官方价格与能力记录量化这一趋势，不直接复用二手预测。
- [Adobe 2026 Creators’ Toolkit Report](https://news.adobe.com/news/2026/06/creators-toolkit-report-2026)显示，75% 的受访创作者认为创意 AI 已融入或成为其工作流的必要部分，但 85% 仍认为最终创意决策应由人完成。这正是 Yoda“AI 执行、人负责驾驭与品味”的市场基础。

以上趋势必须在正式稿中补充近 12 个月的一手或权威来源。

## 4. 市场分析

### 4.1 目标客户

用“个人核心用户—辐射人群—机构客户—长期市场”四层定义，避免在“只服务开发者”与“立即服务所有人”之间摇摆：

1. **核心用户——超级开发者**：能够驾驭一个或多个 Agent，以个人或小团队完成过去需要多人研发团队才能完成的 App、产品和服务。
2. **直接辐射人群——AI 原生创作者与团队**：包括内容创作者、设计师、研究者、运营者、个人创业者，以及广义 Coding 用户。他们主动组合 AI 工具，并需要更可控、连续、可复用的工作方式。
3. **B 端设计伙伴——高校、职业教育/培训机构、企业研发与学习发展部门**：他们需要统一的 AI 实践环境、可复用的课程或内部工作流、清晰的数据与权限边界，以及二次开发、私有部署、培训和持续支持能力。
4. **长期市场——所有以 AI 为生产方式的创造者与组织**：不以是否具备专业编程能力划分，而以是否需要持续驾驭 AI、管理创作过程并交付真实成果划分。

这里的“超级开发者”是核心原型，不是市场边界；“创作者”也不是“开发者”的营销替换词。B 端需求目前来自客户交流中的定性信号，不等于已签合同或已验证收入。正式 BP 一方面必须用不同个人用户能否完成首次作品、创作类型占比和跨人群留存证明 C 端辐射路径，另一方面要用付费试点、预算负责人、采购流程、部署要求和续约意向验证 B 端路径。

### 4.2 “碗—锅—田”的市场演进

| 层级 | 用户与需求 | Yoda 的产品形态 | 需要证明的指标 |
| --- | --- | --- | --- |
| 碗：个人超级开发者 | 一个人驾驭 AI 完成真实产品 | Desktop + Harness + Relay / Creator Pro | 激活、作品完成、周留存、个人付费 |
| 锅：AI 原生团队与教育/培训机构 | 多人共享上下文、方法、权限和成果；统一开展教学、实训或内部培训 | Studio / Education Edition + 共享 Skills/Memory + 课程模板、协作与治理 | 席位扩张、团队留存、付费试点转年约、工作流复用 |
| 田：AI 原生组织 | 少数人类与大量 AI 执行能力共同工作 | 跨模型的组织级 Harness 与创作操作系统 | 企业合同、组织渗透、治理与合规 |

“未来组织会由少数人类与大量 AI 执行能力组成”是战略判断，不是已验证事实。Yoda 当前的碗是锋利的个人 Harness；高校教育可以成为较早出现的 B 端切口，通过 Studio / Education Edition 设计伙伴验证标准化交付；田则是长期成为 AI 原生组织创作基础设施的方向。这样既保留百亿级想象空间，也不把尚未交付的机构能力写成现状。

### 4.3 TAM / SAM / SOM 基准测算

| 市场 | 基准数字 | 计算方式 | 口径说明 |
| --- | ---: | --- | --- |
| TAM | 248 亿美元/年 | 2.07 亿全球创作者 × 120 美元年费 | 创作者人数来自 Visa 2025 报告；假设每位创作者都可能购买一套基础 AI 创作工具，因此是理论上限 |
| SAM | 24.8 亿美元/年 | 2.07 亿 × 10% AI 原生创作者适配率 × 120 美元年费 | 10% 是对超级开发者及近期会主动使用 Harness 创作产品或内容人群的待验证假设 |
| SOM（3 年） | 240 万美元 ARR | 2 万个付费账户 × 120 美元年费 | 是经营目标，不是当前收入或市场份额事实 |

- [Visa 2025 Creator Report](https://corporate.visa.com/en/solutions/commercial-solutions/knowledge-hub/2025-visa-creators-report.html)估计全球约有 2.07 亿创作者，并预计创作者经济到 2027 年达到约 5,000 亿美元。该规模只用于说明宏观背景，不直接当作 Yoda 的软件收入 TAM。
- [SlashData](https://www.slashdata.co/research/developer-population)估计全球有 4,840 万活跃开发者。这部分人群不再定义 Yoda 的 TAM，而是 Yoda 当前产品最容易触达和验证价值的首个滩头市场。
- 120 美元年费与当前 Relay 990 credits/30 天（约 9.9 美元/月）的价格锚点接近，正式模型仍需区分免费版、Creator Pro、Relay 和 Studio。
- TAM 假设所有创作者都可能购买基础 AI 创作工具，SAM 又假设其中 10% 属于近期适配人群；两者都必须通过用户研究、定价测试和真实转化继续缩小误差，不能直接当收入机会。
- 当前 TAM/SAM/SOM 只测算个人创作者工具市场，没有把高校和企业采购重复叠加。B 端市场应在获得首批试点报价后，按“目标机构数 × 可服务部门/课程数 × 年均合同金额”自下而上测算并单列。

### 4.4 市场趋势

- **创造民主化**：自然语言逐步成为代码、产品和内容的新创作界面，非技术人群首次能够直接参与软件创造。
- **从助手到可驾驭的工作系统**：Agent 从回答问题升级为承担任务，人需要的产品从单个聊天框转向统一上下文、过程观察、干预和结果验收。
- **创作者即企业**：创作者从内容发布者扩展为独立产品经营者，需要覆盖生产、分发、变现和跨设备协作的基础设施。
- **可复用的创作资产**：Skills、Hooks、Memory 和 Agent 工作流开始成为可组合、可分发的数字生产资料。
- **高校开始采购机构级 AI 工作环境**：[OpenAI 的 ChatGPT Edu](https://openai.com/index/introducing-chatgpt-edu/)提供高校工作区、共享定制能力、权限与管理控制；[Anthropic 的 Claude for Education](https://www.anthropic.com/news/introducing-claude-for-education?lang=us)通过校园合作、学习模式和培训项目进入高校；[GitHub Education](https://github.com/education/students)则长期以统一开发环境、专业工具和教学资源服务学生与教师。这些案例说明高校会为“统一工具 + 管理治理 + 教学推广”采购，但不能直接证明高校会购买 Yoda，仍需通过付费试点验证。

## 5. 竞争格局与竞争楔子

### 5.1 建议比较的五类方案

- 单一厂商 Agent IDE / CLI。
- 面向非技术用户的 AI App Builder 与无代码平台。
- GPT、Claude 等通用 AI 助手和 AIGC 创作工具。
- 云端自主编码 Agent。
- 多 Agent / Agent 团队编排框架或工作台。

正式稿选取 3–5 个真实竞品，按下列维度建立矩阵：创作者门槛、输出类型、供应商中立、单/多 Agent、创作范式、会话恢复、Harness 可观测、结果验证、移动/远程、团队协作、价格。

多 Agent 或 Agent 团队产品不是 Yoda 的类别定义，只是部分能力重叠。Yoda 的竞争边界是“谁能帮助用户更完整地驾驭 AI 创作过程”，而不是“谁能同时启动更多 Agent”。

### 5.2 Yoda 的竞争楔子

- **供应商中立**：统一承载不同 Agent CLI，用户可按任务选择工具并保留资产。
- **Harness 可驾驭**：不只管理聊天窗口或 Agent 数量，而是让上下文、Skills、Hooks、Memory、权限、过程和结果可以被人理解、控制和复用。
- **从想法到真实作品**：当前先把 App 创作中的会话、Diff、Review、CI、发布和继续迭代连接起来，再扩展到更多数字创作流程。
- **本地与云端混合架构**：项目和代码默认留在用户设备，模型、MaaS、同步与 Relay 等云能力按需接入，边界清楚且不误导为“本地大模型”。
- **人的品味始终在环**：Yoda 不追求让 AI 完全替代创作者，而是让人持续掌握目标、选择、干预和最终决定。

### 5.3 技术壁垒与 Harness 资产

正式 BP 增加一张投资人能看懂的核心架构图：

```text
模型 / MaaS
  ↓
31 种 Agent Client 统一适配层
  ↓
Yoda Harness：Session / Skills / Hooks / Memory / 权限 / 状态
  ↓
执行与隔离：单 Agent / 多 Agent / Worktree / 远程 / Relay
  ↓
结果验证：Diff / Review / Test / CI/CD / Release
  ↓
可继续迭代的 App、作品与组织资产
```

壁垒不只来自“接入数量”，而来自四层积累：

1. **兼容性壁垒**：31 种客户端的命令、事件、会话恢复和状态语义被统一到稳定适配层。
2. **流程壁垒**：从任务、会话、隔离到 Review、CI 和发布形成完整状态模型，而非单点功能拼接。
3. **资产壁垒**：用户持续沉淀 Prompt、Skills、Hooks、Memory、模板和工作流，复用价值随使用增加，形成合理迁移成本。
4. **信任壁垒**：供应商中立、清晰的数据边界、跨平台交付和真实自用验证（dogfooding）共同建立长期信任。

“数据飞轮”必须谨慎表述：用户代码和 Harness 资产默认不是 Yoda 可任意训练的中心化数据。可成立的飞轮来自经用户授权的匿名产品信号、公开生态资产、兼容性经验与社区贡献。

技术证据页需补充：统一适配一个新 Agent 所需时间；并发会话下的 CPU、内存和启动耗时；Worktree/状态模型减少冲突的对照；会话异常后的恢复成功率。未实测前只列验证计划，不写性能优势结论。

### 5.4 诚实披露的竞品优势

- 单一厂商产品可能拥有更深的模型集成、默认体验和资金优势。
- 云端 Agent 可能拥有更强的托管算力和无人值守体验。
- IDE 类产品拥有成熟编辑器生态和庞大分发渠道。
- Yoda 需要证明跨客户端编排带来的价值足以抵消配置和学习成本。

## 6. 商业模式与收入模型

### 6.1 C 端与 B 端双轨商业路径

```text
C 端：开源 Yoda Desktop 建立超级开发者口碑
  → Relay / Creator Pro 验证个人付费
  → Studio 与 Marketplace 扩展团队协作和生态收入

B 端：以开源、可二次开发的 Yoda 作为机构采用底座
  → 通过付费试点验证高校教学、内部培训和研发实践场景
  → 推出 Education / Organization Edition 标准产品
  → 形成机构年费、部署治理、培训实施与持续服务收入
```

两条路径共用同一套 Harness 产品底座。C 端贡献口碑、活跃用户和产品反馈；B 端提供更高客单价、更稳定的年度合同和组织级需求验证。B 端不是把 Yoda 改造成一次性交付的软件项目，而是把可复用的组织能力产品化。

### 6.2 收入流

**C 端及团队收入（按验证顺序）**

1. **Relay Pass / 创作者订阅**：跨网络安全连接桌面 Agent，验证个人创作者持续付费意愿。
2. **Creator Pro 云能力**：云同步、跨设备创作、增强自动化、作品历史和发布能力；范围与价格待验证。
3. **Studio 席位**：工作室权限、共享 Agent/Skills/Memory、审计、品牌资产与协作工作流。
4. **Marketplace 抽成**：第三方 Skills、模板和工作流交易；必须在供需两侧形成规模后启动。

**B 端机构收入（按验证顺序）**

1. **付费试点**：围绕一门课程、一期训练营或一个内部培训 cohort，以明确周期、人数、验收指标和交付边界验证需求。
2. **Education / Organization Edition 年度订阅或服务合同**：按机构、校区、站点或活跃席位收费，提供官方维护版本、统一配置、共享 Harness 资产、组织管理和版本升级。
3. **部署与治理服务**：私有化或专属云部署、模型/MaaS 接入、账号权限、审计、安全策略和现有系统集成；当前均属于待验证/待产品化能力，不写成已交付现状。
4. **培训与课程服务**：教师/助教培训、内部讲师培训、标准课程包、实训模板、认证和上线辅导。
5. **二次开发、OEM 与持续支持**：对可复用需求提供付费扩展、年度维护和 SLA；高度专属需求单独按专业服务收费并设置交付上限。

Yoda 采用 Apache-2.0，机构无需付费即可 Fork 和修改源代码。因此，B 端不能把“允许二次开发”本身当作收费点；真正可持续的价值来自官方维护版本、标准化机构能力、部署治理、课程与培训、集成效率、升级兼容、响应 SLA 和长期支持。

### 6.3 高校教育作为首个 B 端切口

高校教育与 Yoda 的匹配点不只是“学生学编程”，而是学校需要一套能够统一承载不同模型、Agent 和教学方法的 AI 实践环境：

- **统一实践环境**：降低不同电脑、Agent Client、模型账号和配置造成的教学摩擦。
- **可复制的教学资产**：教师把 Skills、Hooks、Memory、Prompt、模板和参考工作流配置成课程包，学生在统一基线下实践。
- **全过程教学与验收**：支持教师演示、学生创作、过程观察、Diff/Review、测试与成果验收，而不只查看最终答案。
- **受控的模型与数据边界**：按学校要求接入指定 MaaS/模型服务，并逐步提供账号、权限、审计和部署选项。
- **二次开发能力**：学校、实验室或合作伙伴可基于开源底座增加校内课程、系统集成和品牌化能力。

建议的首个标准产品包为 **Yoda Education（暂定名）**：Yoda Desktop + 机构配置与课程模板 + 组织管理能力 + 部署选项 + 教师/助教培训 + 年度升级与支持。首轮不追求覆盖全校，而是从一门课程、一个实验室或一期训练营的付费试点开始。

产品化原则：至少 80% 的交付来自共享产品与标准包，最多 20% 为机构配置；无法被三个及以上潜在客户复用的需求，进入单独报价的专业服务或暂不承接。用交付人天、服务毛利和专属代码分支数量约束定制范围。

### 6.4 底层收入公式

- 月新增付费用户 = 有效触达 × 安装率 × 激活率 × 试用转付费率。
- 月经常性收入 = 各套餐活跃付费用户 × 对应月均客单价。
- LTV = 月均毛利 × 平均付费生命周期。
- CAC = 获客相关现金支出 ÷ 同期新增付费用户。
- 回本周期 = CAC ÷ 单用户月均毛利。
- B 端 ARR = 活跃签约机构数 × 年均合同金额（ACV）。
- 试点转年约率 = 签署年度合同的机构数 ÷ 已完成付费试点的机构数。
- 服务毛利率 =（服务收入 − 直接交付人力与第三方成本）÷ 服务收入。

正式稿需明确：免费与付费边界、套餐价格、退款/续费规则、支付成本、Relay 基础设施成本、机构 ACV、试点范围、交付成本，以及开源核心与商业版本的边界。

## 7. 运营计划

### 7.1 产品交付

- Desktop 通过 GitHub Releases 和官网安装包分发。
- 版本更新、签名、公证、跨平台构建和镜像分发由 CI/CD 承担。
- 用户代码与本地状态默认留在本机；Relay、账号和云同步明确区分数据边界。
- 文档、Issue、更新日志和社区内容共同承担上手、支持与反馈收集。
- B 端采用“需求诊断 → 标准付费试点 → 部署与培训 → 验收 → 年约与续费”的交付流程；产品订阅、专业服务和第三方成本分别报价与核算。

### 7.2 一人公司关键流程

1. 用户反馈与问题归类。
2. Feature → Task → Agent 执行 → Review → Verification → Release。
3. 跨平台打包、更新源与公开资产验证。
4. Relay/云服务可用性、成本与安全监控。
5. 内容发布、案例沉淀和用户访谈。

### 7.3 自动化与外包边界

- 优先自动化测试、构建、发布、更新源验证、支持分类和经营数据看板。
- 当支持工时、平台兼容性或内容生产超过预设阈值时，引入兼职/外包伙伴。
- 当标准 B 端交付连续在三个以上客户复用后，再认证教育实施或系统集成伙伴；在此之前由创始人直接参与需求和验收，避免过早扩张渠道。
- 产品方向、架构、安全边界和关键用户访谈继续由创始人直接负责。

## 8. 营销与销售计划

### 8.1 定位语

- **产品定位**：Yoda 是超级开发者的 AI Harness 工作台。
- **功能副标题**：统一 Agent、上下文与交付，让 AI 真正进入持续创作。
- **品牌愿景**：让每个创造者都能驾驭 AI，自由创造。
- **认知对标**：如果说 IDE 统一了代码、调试与构建，Yoda 统一的就是 Agent、上下文与交付。

无论用户使用单个 Agent 还是多个 Agent，Yoda 都让上下文、能力、过程与结果保持可观察、可控制、可复用。

### 8.2 优先渠道

1. **超级开发者与 AI 原生社区**：用“一个人驾驭 AI 完成真实产品”的案例触达独立开发者、个人创业者和小型 AI 原生团队。
2. **Yoda 开发 Yoda 的公开案例**：持续发布真实任务从 Issue、Agent 执行、Review 到 Release 的过程，用产品自身的交付速度证明 Harness 价值。
3. **内容与技术创作者社区**：通过作品案例、模板、GitHub 和可复现工作流，将 Harness 能力翻译成不同创作人群可理解的收益。
4. **创始人内容与生态合作**：围绕一人公司、Agent 创作和 Build in Public 持续输出，并与模型平台、MaaS、Skill 作者和创作者平台联合推广。
5. **高校、商学院与孵化器设计伙伴**：利用已有分享和社区触点，寻找愿意从一门课程、一个实验室、一期训练营或一个内部培训项目开始付费试点的机构；现有关系只代表潜在线索，不写成已签客户。

### 8.3 销售漏斗

```text
内容/社区触达
  → 官网或 GitHub
  → 下载与首次启动
  → 完成并保存首个真实作品
  → 7 日内继续迭代或完成第二个作品
  → Relay/Pro 试用
  → 付费与续费
```

正式稿为每一层补充当前基线、12 个月目标、责任渠道与实验计划。北极星指标建议验证“每周借助 Yoda 完成并被创作者验收的作品数”，而不是只看下载、Stars、Agent 数量或调用次数。

### 8.4 B 端销售漏斗

```text
客户访谈与场景诊断
  → 明确预算负责人、IT/安全参与者和验收指标
  → 一门课程 / 一期训练营 / 一个部门的付费试点
  → 激活率、课程完成率、作品验收率、教师/管理员工时评估
  → Education / Organization Edition 年度合同
  → 扩展到更多课程、学院、校区或部门并续约
```

高校早期买方可能包括学院/实验室负责人、人工智能或计算机相关院系、继续教育与创新创业部门；IT 与安全部门通常影响部署和采购。企业侧优先接触研发效能、AI 平台和学习发展负责人。以上角色均需在访谈中逐一验证，不凭产品团队想象确定。

首阶段目标不是免费做大量定制，而是找到 2–3 家愿意共同定义标准包、支付试点费用并提供验收反馈的设计伙伴。免费概念验证只在范围极小、具有明确决策人和付费转化日期时例外。

## 9. 财务预测

### 9.1 模型结构

- 第 1 年按月，第 2–3 年按季度。
- 至少包含保守和基准情景；乐观情景单列，不混入经营承诺。
- 收入：Relay/Creator Pro、Studio、Marketplace、Education/Organization 年约、部署治理、培训实施、定制扩展与持续支持分别建模。
- 成本：支付手续费、Relay/云资源、监控、代码签名/发布、SaaS 工具、市场投放、外包、人员，以及 B 端售前、部署、培训和支持工时。
- 输出：收入、直接成本、毛利、运营费用、净利润/亏损和期末现金。

### 9.2 必算阈值

- 月度盈亏平衡所需付费用户数。
- Relay/云能力的单用户边际成本与毛利率。
- 创始人投入和现金储备对应的可持续月数。
- 在付费转化、续费率或基础设施成本恶化时的现金流敏感性。
- B 端试点的直接交付成本、转年约率、年均合同金额和回款周期。
- 产品订阅毛利与专业服务毛利分开核算，防止服务收入掩盖产品不可复制。

## 10. 风险评估

正式稿按“概率 × 影响 × 缓解措施 × 预警指标”维护：

| 风险 | 概率 | 影响 | 缓解措施 | 预警指标 |
| --- | --- | --- | --- | --- |
| 单一创始人依赖与倦怠 | 中高 | 高 | 自动化发布与支持；关键流程文档化；建立兼职伙伴池 | 交付延期、支持积压、连续高工时 |
| Agent CLI/API 高频变化 | 高 | 高 | 可扩展 registry；兼容性测试；按使用量划分维护优先级 | 集成故障率、修复时长 |
| 大厂把编排能力内置 | 高 | 高 | 坚持供应商中立、清晰数据边界和跨客户端 Harness 资产层 | 跨客户端用户占比下降 |
| 用户认可产品但不付费 | 中高 | 高 | 先验证 Relay/Pro 的刚需与续费，再扩展商业范围 | 试用转付费、续费率低 |
| 开源与付费边界失衡 | 中 | 高 | 明确核心开源、云连接/团队治理付费的原则 | 社区反弹、付费功能使用低 |
| 安全、隐私或供应链事件 | 中 | 高 | 最小权限、本地默认、签名公证、依赖审计、事件响应流程 | 异常访问、签名/更新失败 |
| 多平台与 31 客户端维护成本过高 | 高 | 中高 | 按活跃使用分级支持；自动化回归；社区共建 | 平台缺陷积压、兼容性 SLA 下降 |
| Marketplace 冷启动 | 中高 | 中 | 在交易前先做可分享目录、官方模板和作者激励 | 有效供给、安装与复用率低 |
| B 端定制拖入项目制 | 中高 | 高 | 标准产品包；80/20 交付原则；专属需求单独报价；设置服务毛利和交付人天门槛 | 专属代码分支、单客户工时、服务毛利恶化 |
| 高校采购周期长、多角色决策 | 中 | 中高 | 从有预算负责人的小范围付费试点切入；提前绘制决策链、采购日历和安全清单 | 免费试点延期、无明确决策人、回款周期拉长 |

## 11. 18–24 个月季度里程碑

以下是投资人版本的建议目标，用户数和收入目标需与完整财务模型再次校准：

| 时间 | 核心目标 | 关键交付与验证 |
| --- | --- | --- |
| 0–3 个月 | 产品功能完善、体验顺滑 | 打磨单 Agent、多 Agent、Vibe Coding 与 Build 主路径；完善 Harness 的观察、干预和复用体验；建立激活、作品完成、DAU、D7/D30 留存埋点；完成首轮 20 位跨类型用户访谈 |
| 4–6 个月 | 建立自然增长并验证 B 端需求 | 完善官网、创作者模板和真实作品案例；验证“首次完成真实作品”激活漏斗；访谈 10 家高校/培训/企业潜在客户，筛选 2–3 家有预算和决策人的设计伙伴，定义标准付费试点 |
| 7–9 个月 | 开始双轨付费验证 | 小规模投放并迭代 Relay/Pro 定价；启动 2–3 个课程、训练营或内部培训付费试点，记录激活、完成、验收、教师/管理员工时和直接交付成本 |
| 10–12 个月 | 找到高留存场景并产品化机构共性 | 聚焦 2–3 个个人高留存场景；从试点共性中形成 Yoda Education / Organization Edition 最小标准包，验证至少一个试点转年度合同 |
| 13–15 个月 | 验证 Studio 与机构年约 | 与设计伙伴测试共享 Agent、Skills、Memory、课程/品牌资产、权限和审计；分别形成 Studio 与 Education/Organization 的最小商业闭环 |
| 16–18 个月 | 扩大市场推广 | 加大有效渠道投放和生态合作；扩展英文市场；验证创作者订阅与 Studio 席位的组合收入 |
| 19–21 个月 | 建立 Harness 生态与交付伙伴网络 | 完善 Skill/工作流的发布、信任、安全扫描和发现；先验证供给、安装与复用，再决定交易抽成；仅在标准 B 端交付已可复制时引入教育实施或集成伙伴 |
| 22–24 个月 | 证明规模化能力 | 形成个人付费账户与机构年约相互增强的增长模型；C 端继续检验冲击 2 万付费账户、240 万美元 ARR 的三年 SOM，B 端收入单列且不重复计算；验证增长效率、毛利和组织可持续性是否支持下一轮融资 |

每季度设置继续、调整或停止的量化门槛。前三个月不急于大规模投放，先把产品体验和数据基础做扎实；之后再根据真实留存逐步增加推广预算。
